@thecaticorn01 — good question, and you're right that the original explanation was unclear.

**How they're meant to coexist:**
Both apps can run at the same time — they each start a separate privileged server process (`shizuku_server` vs `shizuku_plus_server`) and register separate binders. Starting one does not stop the other.

**Why apps don't "see" Shizuku+ automatically:**
Third-party apps (SD Maid SE, SwiftBackup, etc.) discover Shizuku by connecting to `moe.shizuku.privileged.api` — that's the stock Shizuku package name hard-coded in their binding logic. Shizuku+ has a different package name (`af.shizuku.plus.api`), so those apps will never find it through normal discovery. **This isn't a runtime conflict — they're just looking at different packages.**

**What "extended features" actually means:**
Right now, the "extended features" Shizuku+ provides are accessed through the Shizuku+ manager UI itself (Overlay Manager Plus, AICore Bridge, Root Compatibility Hub, etc.) — not through existing third-party apps. A third-party app would have to be explicitly coded to use the Shizuku+ API to benefit from those features.

**Practical setup:**
- Start stock Shizuku → grant permissions to SD Maid SE, SwiftBackup, MTManager, etc. inside Shizuku's app list. Those apps work normally.
- Start Shizuku+ → use its manager features (OMP, Watchdog, activity log, etc.).
- They run in parallel without interfering.

The "override" behaviour you saw may have been apps not finding stock Shizuku's binder when only Shizuku+ was running — which is expected, since they're looking for a different package. Starting both resolves it.
