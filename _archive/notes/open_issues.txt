Thanks for the thorough write-up — your hypothesis about the package name is correct.

**Why apps don't detect Shizuku+:**
Apps bind to Shizuku using the package name `moe.shizuku.privileged.api`. Shizuku+ uses `af.shizuku.manager`, so apps that hard-code the original package name won't discover it automatically. This is a known compatibility gap. The recommended workaround for now is to keep stock Shizuku installed alongside Shizuku+ if you need broad third-party app support — Shizuku+ handles the extended features while stock Shizuku handles discovery.

**Service timeout (30s TimeoutException):**
On Android 16 / OneUI 8.0 the ADB handshake timing has changed. We're aware of this and investigating. As a workaround, try starting the service immediately after enabling Wireless Debugging (before the system applies any background restrictions).

We'll track both issues in #199.

@thecaticorn01 — good question, and you're right that the original explanation was unclear.

**How they're meant to coexist:**
Both apps can run at the same time — they each start a separate privileged server process (`shizuku_server` vs `shizuku_plus_server`) and register separate binders. Starting one does not stop the other.

**Why apps don't "see" Shizuku+ automatically:**
Third-party apps (SD Maid SE, SwiftBackup, etc.) discover Shizuku by connecting to `moe.shizuku.privileged.api` — that's the stock Shizuku package name hard-coded in their binding logic. Shizuku+ has a different package name (`af.shizuku.plus.api`), so those apps will never find it through normal discovery. **This isn't a runtime conflict — they're just looking at different packages.**

**What "extended features" actually means:**
Right now, the "extended features" Shizuku+ provides are accessed through the Shizuku+ manager UI itself (Overlay Manager Plus, AICore Bridge, Root Compatibility Hub, etc.) — not through existing third-party apps. A third-party app would have to be explicitly coded to use the Shizuku+ API to benefit from those features.

**Practical setup:**
- Start stock Shizuku → grant permissions to SD Maid SE, SwiftBackup, MTManager, etc. inside Shizuku's app list. Those apps work normally.
- Start Shizuku+ → use its manager features (OMP, Watchdog, activity log, etc.).
- They run in parallel without interfering.

The "override" behaviour you saw may have been apps not finding stock Shizuku's binder when only Shizuku+ was running — which is expected, since they're looking for a different package. Starting both resolves it.

Thanks for the report. On Android 15, the Pair button opens the system's **Wireless Debugging** pairing screen — if it appears to do nothing, the most common cause is one of these:

1. **Wireless Debugging is not enabled.** Open **Settings → Developer Options → Wireless debugging** and make sure it is toggled on. If Developer Options isn't visible, tap your build number 7 times in Settings → About phone.
2. **Developer Options are disabled between sessions.** Some Android 15 ROMs (including Honor's MagicOS) reset Developer Options after a reboot. Re-enable the toggle, then tap Pair again.
3. **Honor-specific restriction.** MagicOS 9 (Android 15) has been observed to block the `ACTION_WIRELESS_DEBUGGING_PAIR` intent on some builds. If the Pair button opens nothing at all, go to **Developer Options → Wireless debugging → Pair device with pairing code** manually, note the IP/port/code shown, and use the in-app pairing dialog with those values.

If none of these apply, let me know which MagicOS build you're on and whether tapping Pair shows any response (a brief ripple, a dialog, or nothing at all).
